# Product Requirements Document (PRD)
## Система управления производством: ТОО KUS & ТОО SSAP

**Версия:** 1.0  
**Дата:** 18 июня 2026  
**Статус:** Утверждено  
**Язык:** Русский

---

## 1. Обзор продукта

### 1.1 Название
**Система управления производством (Production Management System)**

### 1.2 Назначение
Веб-приложение для ежедневного сбора, отслеживания и анализа производственных показателей двух промышленных предприятий:
- **ТОО KUS** — производство электроэнергии и очистка воды
- **ТОО SSAP** — производство серной кислоты

### 1.3 Основная цель
Обеспечить операторам и менеджерам предприятий удобный инструмент для:
- Ежедневного ввода производственных показателей
- Отслеживания выполнения плановых показателей
- Анализа производительности за выбранные периоды
- Сравнения фактических и плановых значений

### 1.4 Время разработки
Доставка готового продукта: **1 день**

---

## 2. Проблема и решение

### 2.1 Выявленная проблема
- Отсутствие централизованной системы сбора производственных данных
- Данные разрозненны (таблицы Excel, бумажные журналы)
- Сложный анализ и сравнение показателей по разным периодам
- Невозможность оперативного контроля выполнения плана
- Отсутствие единого источника информации для всех пользователей

### 2.2 Предлагаемое решение
Веб-приложение с:
- **Дашбордом** для оперативного ввода данных
- **Отчётностью** с фильтрацией по датам
- **Визуализацией** показателей через графики и таблицы
- **Локальным хранилищем** (браузер) для автономной работы

## 4. Функциональные требования

### 4.1 Модуль "Дашборд ввода данных"

#### 4.1.1 Управление предприятиями
**Требование FR-101**
- Система должна поддерживать два независимых предприятия: KUS и SSAP
- Пользователь должен иметь возможность переключаться между предприятиями через вкладки
- Каждое предприятие имеет свой набор показателей
- Отделение данных по предприятиям в хранилище

#### 4.1.2 Ввод данных для ТОО KUS
**Требование FR-102**
Система должна позволять вводить следующие показатели:
- **Выработка электроэнергии (кВч):**
  - Фактическое значение (обязательное)
  - Плановое значение (обязательное)
  - Формат: числовое, до 2 знаков после запятой
  
- **Объём очищенной воды (м³):**
  - Значение за день (обязательное)
  - Формат: числовое, до 1 знака после запятой
  
- **Качество воды:**
  - Текстовое описание (опционально)
  - Пример: "pH 7.2, мутность в норме"
  - Макс. длина: 500 символов

- **Дата записи:**
  - Выбор календарём (обязательное)
  - По умолчанию: текущая дата
  - Возможность выбирать прошлые даты

#### 4.1.3 Ввод данных для ТОО SSAP
**Требование FR-103**
Система должна позволять вводить следующие показатели:
- **Производство серной кислоты (т):**
  - Фактическое значение (обязательное)
  - Плановое значение (обязательное)
  - Формат: числовое, до 1 знака после запятой
  
- **Концентрация (%):**
  - Значение за день (опционально)
  - Формат: числовое, до 1 знака после запятой
  - Допустимый диапазон: 0-100%
  
- **Качество продукции:**
  - Текстовое описание (опционально)
  - Пример: "без примесей, плотность в норме"
  - Макс. длина: 500 символов

- **Дата записи:**
  - Выбор календарём (обязательное)

#### 4.1.4 Сохранение данных
**Требование FR-104**
- При нажатии "Сохранить данные" система должна:
  - Валидировать обязательные поля
  - Показывать сообщение об ошибке при неполных данных
  - Сохранять данные в localStorage браузера
  - Отобразить успешное сохранение (сообщение + анимация)
  - Очистить форму ввода
  - Обновить статистику на дашборде

#### 4.1.5 Отображение последних показателей
**Требование FR-105**
На дашборде должны отображаться последние показатели:
- **Для KUS:**
  - Последняя выработка электроэнергии (факт и план)
  - Процент выполнения плана
  - Цветовая индикация (зелёный ≥95%, оранжевый <95%)

- **Для SSAP:**
  - Последнее производство серной кислоты (факт и план)
  - Процент выполнения плана
  - Цветовая индикация

#### 4.1.6 История записей на дашборде
**Требование FR-106**
- Система должна отображать таблицу с последними 10 записями
- Колонки: Дата, Компания, Факт, План, %, Примечание
- Сортировка: по дате (новые сверху)
- Строки с процентом ≥95% — подсвечены зелёным
- Строки с процентом <95% — подсвечены оранжевым

### 4.2 Модуль "Общий отчёт и анализ"

#### 4.2.1 Фильтрация по датам
**Требование FR-201**
- Система должна предоставлять два поля выбора дат:
  - "Дата начала" (обязательное)
  - "Дата окончания" (обязательное)
- По умолчанию: минимальная и максимальная даты имеющихся данных
- Кнопка "Применить фильтр" для обновления отчёта
- Кнопка "Сбросить фильтр" для возврата к полному диапазону
- Отображение информации: "Показано X записей за Y дней"

#### 4.2.2 Итоговая статистика за период
**Требование FR-202**
После применения фильтра отображать:
- **Для KUS:**
  - Суммарная выработка электроэнергии (факт)
  - Суммарный план выработки
  - Процент выполнения плана за период
  - Количество дней с данными

- **Для SSAP:**
  - Суммарное производство серной кислоты (факт)
  - Суммарный план производства
  - Процент выполнения плана за период
  - Количество дней с данными

- **Визуализация:**
  - 6 карточек (метрик) в сетке
  - Цветовая индикация в зависимости от % выполнения

#### 4.2.3 График выполнения плана
**Требование FR-203**
- Система должна отображать линейный график с:
  - **Ось X:** даты (в формате ДД.ММ.ГГГГ)
  - **Ось Y:** процент выполнения плана (0-120%)
  - **Две линии:**
    - KUS (синий цвет #185FA5)
    - SSAP (оранжевый цвет #D85A30)
  - **Пунктирная линия:** целевое значение 95%
  - Интерактивность: tooltip при наведении
  - Легенда с названиями линий

#### 4.2.4 График сравнения производства
**Требование FR-204**
- Система должна отображать столбчатый график с двойной осью Y:
  - **Левая ось Y:** KUS (кВч)
  - **Правая ось Y:** SSAP (т)
  - **Столбцы:**
    - KUS факт (синий #185FA5)
    - KUS план (светло-синий #B5D4F4)
    - SSAP факт (оранжевый #D85A30)
    - SSAP план (светло-оранжевый #F0997B)
  - Интерактивность: tooltip при наведении
  - Легенда с названиями

#### 4.2.5 Таблица записей за период
**Требование FR-205**
- Система должна отображать таблицу со всеми записями за выбранный период
- Колонки: Дата, Компания, Факт, План, %, Примечание
- Сортировка: по дате (новые сверху)
- Количество показываемых записей: все записи за период (скролл)
- Цветовая индикация:
  - Зелёный текст для % ≥95%
  - Оранжевый текст для % <95%
- Чередующийся фон строк (серая/белая)

#### 4.2.6 Консистентность данных между модулями
**Требование FR-206**
- Изменения в дашборде должны немедленно отражаться в отчёте
- Данные, добавленные в дашборд, должны появляться в таблице отчёта
- Фильтры в отчёте должны работать только с сохранёнными данными

## 5. Нефункциональные требования

### 5.1 Производительность (NFR-101)
- Время загрузки страницы: < 3 сек на медленном интернете
- Отклик при нажатии кнопки: < 500 мс
- Обновление графиков: < 1 сек
- Поддержка до 10,000 записей без заметного замедления

### 5.2 Доступность (NFR-102)
- Полная функциональность на браузерах:
  - Chrome 90+
  - Firefox 88+
  - Safari 14+
  - Edge 90+
- Поддержка мобильных браузеров (адаптивный дизайн)
- Работа на разрешениях от 320px до 2560px

### 5.3 Безопасность (NFR-103)
- Локальное хранилище (localStorage) браузера — не требует аутентификации
- Данные хранятся только на компьютере пользователя
- Шифрование в локальном хранилище: не требуется (данные локальны)
- Защита от XSS: все пользовательские данные экранируются

### 5.4 Надёжность (NFR-104)
- Восстановление после ошибок:
  - При ошибке сохранения показать сообщение пользователю
  - Повторное сохранение с валидацией
- Сохранение состояния: автосохранение в localStorage
- Резервное копирование: экспорт данных

### 5.5 Масштабируемость (NFR-105)
- Архитектура позволяет добавить 3-е и последующие предприятия
- Возможность расширения до облачной БД
- Возможность добавления новых показателей без переписывания кода

### 5.6 Удобство использования (NFR-106)
- Интуитивный интерфейс для некомпьютерных пользователей
- Минимум кликов для ввода данных (макс. 5-6 полей)
- Справочные подсказки при наведении
- Цветовая система (зелёный = хорошо, оранжевый = внимание)
- Все тексты на русском языке

### 5.7 Совместимость (NFR-107)
- Формат данных: JSON для локального хранилища
- Возможность экспорта в CSV/Excel
- Возможность импорта из других систем (будущая версия)

## 7. Пользовательские сценарии

### 7.1 Сценарий 1: Ежедневный ввод данных (KUS)
**Актор:** Оператор ТОО KUS  
**Частота:** Ежедневно в 17:00  
**Длительность:** 2-3 минуты

**Шаги:**
1. Открыть приложение (закладка браузера)
2. Убедиться, что выбрана вкладка "ТОО KUS" на дашборде
3. Дата автоматически установлена на сегодня
4. Ввести значения:
   - Выработка (факт): 1250 кВч
   - Выработка (план): 1300 кВч
   - Вода очищена: 850 м³
   - Качество: "pH норм, мутность <1 NTU"
5. Нажать "Сохранить данные"
6. Увидеть сообщение об успехе
7. Проверить статистику обновилась (новый % выполнения плана)

**Результат:** Данные сохранены и видны в отчёте

---

### 7.2 Сценарий 2: Анализ производительности за неделю
**Актор:** Менеджер производства  
**Частота:** Еженедельно (понедельник)  
**Длительность:** 10-15 минут

**Шаги:**
1. Открыть приложение
2. Перейти на вкладку "Общий отчёт и анализ"
3. Выбрать дату начала: 11 июня (пн)
4. Выбрать дату окончания: 17 июня (вс)
5. Нажать "Применить фильтр"
6. Увидеть итоговую статистику за неделю
7. Проанализировать графики:
   - Выполнение плана по дням
   - Сравнение объёмов KUS и SSAP
8. Посмотреть таблицу всех записей за период
9. Если нужны данные за другой период, повторить фильтрацию

**Результат:** Получен анализ производительности за период

---

### 7.3 Сценарий 3: Проверка качества данных
**Актор:** ИТ-специалист или администратор  
**Частота:** По необходимости  

**Шаги:**
1. Открыть приложение
2. На дашборде проверить последние записи
3. В отчёте выбрать нужный период
4. Посмотреть таблицу и убедиться, что данные корректны
5. При необходимости связаться с оператором для уточнения
6. В будущих версиях: экспортировать данные для архива

**Результат:** Верификация целостности данных

## 13. Риски и ограничения

### 13.1 Известные риски
| Риск | Вероятность | Влияние | Решение |
|------|-----------|--------|--------|
| Потеря данных при очистке кэша | Средняя | Критичное | Регулярный экспорт |
| Конфликты данных (оффлайн) | Низкая | Среднее | Синхронизация при подключении |
| Ограничение localStorage (5-10 МБ) | Низкая | Среднее | Архивирование старых данных |
| Нестабильный интернет | Средняя | Низкое | Работает оффлайн |

### 13.2 Ограничения текущей версии
1. **Локальное хранилище только** — нет синхронизации между компьютерами
2. **Одного браузера на компьютер** — данные разделены по браузерам
3. **Нет пользовательских ролей** — все видят все данные
4. **Нет аутентификации** — нет контроля доступа
5. **Максимум 10,000 записей** — при большем объёме требуется облако
6. **Нет поддержки Kazakh** — только русский язык

### 13.3 Предложения по снижению рисков
- ✅ Добавить функцию экспорта в Excel (резервная копия)
- ✅ Добавить сообщение о синхронизации в облако (версия 2.0)
